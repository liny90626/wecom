# OpenClaw WeCom 插件 v2.5.110-112 变更简报

> [!NOTE]
> `v2.5.110-112` 是 LinKy fork 维护版，基于原作者 **YanHaidao** 的 `v2.5.110` 继续做 OpenClaw/企业微信实际使用场景下的兼容性修复、回归验证和文档整理。本版本仅用于个人学习、问题复现和修复验证，不代表原作者发布版本。

## 2026-06-27（v2.5.110-112）

### B1：Markdown 表格渲染修复

- 修复企业微信侧 Markdown 表格无法按表格结构展示的问题。
- 保留表格块的结构化语义，避免在普通 Markdown 转换和长文本分段时被拆散成不可读文本。
- 增加 `scripts/patch-wecom-markdown-table.mjs --check` 作为静态自检入口，后续修改投递链路时需要保护该能力。

### B2：长文本分段、去重和截断防护

- 保持企业微信 Bot WS stream 气泡上限为 `12,000 bytes`，避免盲目放宽限制引入客户端截断。
- 最终正文超过流式气泡限制时，首段通过 `replyStream(..., finish=true)` 关闭原 stream 气泡，后续段落通过 Bot WS 主动推送继续发送。
- 增加 final/preview 去重逻辑，处理 OpenClaw 流式预览和最终 final 文本高度重合时的重复内容。
- 对长报告中“从同一标题重新开始”的重复尾段做结构化去重，同时保护 Markdown 表格、代码块等不应被误删的内容。
- 增加短文本、中等文本和长文本场景的回归用例，避免只有超长文本才触发去重保护。

### B3：长任务、新消息合并和流式窗口兜底

- 新消息到达并合并思考时，只在旧气泡仍处于占位符阶段时发送“已收到新消息，合并思考。✅”提示。
- 如果旧气泡已经输出可见正文，不再用合并提示覆盖已输出内容。
- 对长任务预览增加冻结策略：默认 `5 分钟 / 3,000 字` 后停止继续刷新大段正文，只保留原预览文本并定期更新状态行。
- 当原 stream 更新窗口过期、ack timeout 或 req_id 失效时，最终结果走主动推送兜底，降低“后台已完成但企业微信未触达”的概率。
- 当用户在旧消息输出过程中发送新消息时，旧 final 不再额外主动推送一份完整重复内容。

### Reasoning 思考块预览实验

- Bot WS 回复默认开启 `reasoningPreviewEnabled: true`，并接入 OpenClaw 的 `onReasoningStream` / `onReasoningEnd` 回调。
- reasoning 内容只进入进度流预览，正文 final 保持普通正文路径，减少“思考块和正文块内容完全一致”的风险。
- 进度流中尝试使用企业微信客户端可识别的 `<think>...</think>` 结构承载思考块。
- 普通正文中的字面量 `<think>` 标签会被转义，避免用户正文被误识别为思考块。
- 当前能力仍依赖模型服务是否透传 reasoning 内容、OpenClaw runtime 版本以及企业微信客户端渲染策略，因此标记为实验性能力。

## 已知边界

- Reasoning 思考块不是纯 Markdown 能力，实际展示取决于企业微信 Bot WS stream 渲染和上游 reasoning 内容是否透传。
- B2/B3 兜底以“尽量送达最终正文”为优先目标；当消息被新请求 supersede 且包含媒体附件时，旧流里的附件不会自动重新上传。
- 本 fork 的修复目标是兼容性和稳定性验证，不改变原项目的许可边界和原作者署名。
