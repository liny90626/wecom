# OpenClaw WeCom 插件 v2.5.110-136 变更简报

> [!NOTE]
> `v2.5.110-136` 不是在 v135 状态机上继续叠加保护，而是以 `released/2.5.110-118` 为稳定骨架，重新实现实际需要的修复。原 v135 完整状态保留在 `features/v135-stabilization`，便于审计，不直接合并回主线。

## 2026-07-12（v2.5.110-136）

### 回退与重做原因

- v119-v135 期间大量边界保护集中进入 Bot WS 回复状态机，测试覆盖增加，但生产代码复杂度、长任务占用时间和交互时序风险也同步增加。
- v132-v134 曾出现分钟级旧 core 等待、长 init-conflict 重试和熔断状态；这些机制与 OpenClaw 的真实 session 占用并不等价，已经确认会放大延迟或造成持续占用提示。
- 本版本保留 v118 的轻量投递骨架，只重做能够由复现测试证明的正确性修复，每条入站仍只调用一次 OpenClaw。

### OpenClaw 6.11 与会话时序

- OpenClaw 开发/peer 版本对齐 `^2026.6.11`，企微 SDK 精确固定为 `1.0.6`。
- 三条 record-to-dispatch 路径统一等待 OpenClaw 异步 session metadata task：统一运行时、Bot webhook、Agent callback。
- metadata 失败保持非阻断；等待支持 abort，并保留 60 秒硬上限，既避免 Promise 异常时无限挂起，也不误杀 OpenClaw 合法的串行 session 写入。
- Bot WS reply handle 在 prepare 前登记；新消息可向旧 prepare/core 发出 abort，prepare 最长 60 秒；新 dispatch 等待 250ms quiet grace 后继续，但不把这段等待当成旧 core 已退出的强保证。
- 不增加 init-conflict 本地重试、per-peer 熔断器、分钟级 handoff 或外层 OpenClaw 重投。

### 回复完整性与 ACK

- 结构化长正文仅在同一结构化标题重新开始后删除连续、同序、精确的重复尾段；遇到第一行新内容立即停止删除，不处理不同章节中的合法相同业务段落。
- preview 返回 `skipped` 不再记作已显示；pending 期间只保留最新预览，ACK 恢复后仅补发最新版本。
- pending preview 超时后发送一次后台处理提示，不再恢复成无提示静默。
- late final 只补真实未显示的尾部；只有确认已显示预览完整覆盖 final 时才使用“以上预览内容即为完整回复”，不再用该提示掩盖不确定状态。
- final 等待 pending ACK 后重新核对已确认正文，再计算主动补发尾部，避免等待期间晚到的 ACK 导致整段重复。
- 正常流式 final 的首段成功、后续主动分段失败时，从第一个未送达尾段进入现有 20/40/80 秒有限重试；首段或附件已确认可见后，取消或耗尽尾段重试也不会释放 final claim、重发已送达副作用。
- detached 尾段重试按账号、单聊/群聊、peer 和 activation 隔离。已经显示过正文的旧回复会在新消息到来时取消重试；完全不可见的旧 final 保留既有有限重试，优先避免结果永久丢失。
- callback stream final 明确过期/拒绝或 ACK 结果不确定时只切换一次主动推送；主动推送自身结果不确定后不再进入第二轮重发。
- 主动推送在明确发送失败时仍执行 20/40/80 秒有限重试；ACK timeout、发送后 socket 断开或 SDK `reply ... cancelled` 属于投递结果不确定，不自动重发，避免用户收到重复气泡。
- thinking 与状态行共同占用预览预算时，只按实际确认展示的正文前缀裁剪 final；组合气泡同时守住 3500 字符和 12KB 上限。
- 空 final 会关闭已经打开的 placeholder/block-only stream。
- fail 路径同样等待 pending ACK；无法收敛时改走一次主动推送，避免中断提示也要等到用户催促后才刷新。
- final 的 20/40/80 秒兜底重试继续保持 v118 的 detached 模式，不占用 OpenClaw dispatch 生命周期。

### Fast、无正文与 message-tool

- 保留 Fast auto-off/auto-on 进度气泡。
- 正文预览冻结后，Fast auto-off/auto-on 仍会作为保留尾行刷新，且不会污染正文累计或已展示前缀记账。
- Fast auto-off 后必须出现真实正文、auto-on 终态或实际 message-tool 投递；否则明确显示“本次回复投递中断”，不再停在 Fast 气泡。
- zero-output、无当前 dispatch committed delivery 证据的 tool-only、reasoning-only、reasoningEnd-only 和 `failedCounts` 不再被误判为完整正文。
- thinking 进度在字符或字节截断后会清理悬空的 `<--` / `<!--` 标记，避免残片泄漏到气泡尾部。
- Fast callback 与 direct delivery 只交付一次。message-tool 是否完成直接采用 OpenClaw 当前 dispatch 的 committed `onObservedReplyDelivery` / `observedReplyDelivery` 证据，不再维护缺少可靠 `runId` 的 session 级 `message_sent` 序号。
- 较早 final 候选投递失败时，只要同一次 dispatch 后续 final 已成功送达，就不再让早期错误覆盖成功结果；本版本没有额外实现模型候选 generation 状态机。

### Agent 与附件批次

- Agent 发送队列改为每个发送任务最多执行一次；前一任务或日志 bookkeeping 失败不会让当前任务重复发送或被跳过。
- Bot WS final 在发送附件前先建立 intent claim；重复 final 不会重复附件，附件发送期间被新消息 supersede 后也不会继续穿插旧附件和正文。
- Bot webhook 短时间内的 `text + file`、`file + text`、`image + image + text` 保持原始顺序合并到同一批次；Bot WS 和 Agent 入站不使用这套合批逻辑。
- Bot webhook 已经存在 queued 批次时，新媒体只进入 queued，不倒插旧 active 批次。
- queued 批次只在 debounce 短窗口内继续合并；窗口结束后的消息建立下一批，按序等待 active 完成，不会被吸入几分钟前的旧批次。
- Bot webhook 单批最多处理 8 个附件，并共享 `mediaMaxMb` 总字节预算；所有成功完成 session staging 的附件进入 `Attachments` / `MediaPaths`，失败附件写入正文和 `MediaFailures`。
- Bot webhook 合并回执区分成功与失败；账号 reload、批次过期或 runtime 不可用时会结算批次并推进后续队列。

### 依赖安全

- `fast-xml-parser` 升级到 `5.10.0`，修复 v118 版本中的 critical XML 实体处理漏洞。
- `undici` 升级到 `7.28.0`，并在兼容范围内更新 `form-data` / `ws` lock 版本。
- `npm audit --omit=dev` 的生产依赖漏洞为 0。

### 明确未恢复

- synthetic thinking / 伪思考块。
- 30/35 分钟旧 core 等待和 30.7 秒 init-conflict 重试阶梯。
- per-peer 熔断器、suppressed-conflict 计数和占用提示状态机。
- 同步等待 20/40/80 秒 final 重试。
- 依赖标点或标题的模糊“回复是否完成”猜测。

### 已知边界

- 账号 reload 会退休 target、取消尚未执行的批次并拒绝尚在读取 body 的旧请求，但不会强制中止已经 flush 且进入 OpenClaw 的运行。
- 长任务期间可以形成多个已封口的 queued 批次；每批都有短合并窗口、附件数量和媒体字节预算，但普通文字正文仍沿用 v118 的单条消息限制，没有新增跨批总字节上限。
- 结构化尾段去重是严格启发式规则；它只处理同一结构化标题重新开始后的连续精确重复，优先保留无法确定是否重复的正文。
- callback stream final 的 ACK 丢失同样无法判定服务端是否已经展示；本版为避免长任务完全丢失会进行一次跨路径主动推送，因此极端 ACK 丢失仍可能形成一次 callback/主动推送双气泡，但不会继续递归重发。
- 主动推送结果不确定时选择不重发以避免重复气泡，因此在“服务端未收到且 ACK/连接同时丢失”的极端情况下仍可能需要用户重试。
