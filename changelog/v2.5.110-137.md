# OpenClaw WeCom 插件 v2.5.110-137 变更简报

## 2026-07-13（v2.5.110-137）

### 问题与根因

- 当同一 Bot WS 用户的旧长任务仍在多次工具调用过程中，新消息会先中断旧回复并立即启动新的 OpenClaw dispatch。
- v136 使用固定 250ms quiet grace，但旧任务的 abort 清理仍可能在更新 session store。OpenClaw 6.11 的 session 初始化采用 revision 条件提交，连续两次读到过期 revision 后会抛出 `reply session initialization conflicted`。
- 旧回复已经被 supersede，新回复又在 agent run 初始化前失败，因此用户会同时失去旧任务和新指令的最终结果，并看到包含内部 session key 的错误。

### 修复方案

- 正常接管不再固定等待 250ms，新消息可直接开始，避免无冲突场景增加延迟。
- 仅当本次 Bot WS dispatch 确实接管了同账号、同 peer 的旧回复，并收到精确的 session 初始化冲突时，等待 500ms 后重试一次。
- 首次冲突不创建错误气泡；第二次仍冲突才交给正常失败出口。等待可被更新的第三条消息取消，旧请求不会复活。
- 冲突发生在 OpenClaw agent run 和工具执行开始之前，因此该次重试不会重复执行工具。
- dispatcher 与用户提示共用同一个严格错误识别函数，并支持 OpenClaw 通过 `cause` 包装错误的情况。
- 用户侧统一显示“之前任务还在处理中，新指令冲突啦，请等几分钟后再试试”，不再暴露 `WeCom WS reply failed`、agent 名称或 session key；原始错误仍保留在服务日志中。

### 行为边界

- 非 Bot WS 入站不启用该重试。
- 没有接管旧回复时，即使出现初始化冲突，也不在插件层擅自重试。
- 非精确初始化冲突保持原错误处理，不吞错、不扩大重试范围。
- 持续冲突最多重试一次，不恢复 v132-v134 的重试阶梯、分钟级等待或 per-peer 熔断器。

### 验证

- 修复前，确定性回归测试可复现新消息收到初始化冲突且无法完成；修复后同一场景正常交付新任务 final。
- 覆盖正常零延迟接管、瞬时冲突恢复、持续冲突单次报告、第三条消息取消待重试、无接管不重试、嵌套 `cause`、原气泡提示及流窗口失效后的主动推送。
- TypeScript 构建通过；Vitest 全量 `41 files / 342 tests` 通过；B1/B2/B3 均为 `READY`。
