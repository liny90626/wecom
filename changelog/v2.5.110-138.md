# OpenClaw WeCom 插件 v2.5.110-138 变更简报

## 2026-07-15（v2.5.110-138）

### 问题与根因

- Bot WS 会把媒体和文字作为相邻的独立入站帧发送。插件此前在收到媒体帧时立即 dispatch，紧随其后的文字帧因此变成另一条入站，OpenClaw 只拿到文件上下文，用户必须再次发送文字。
- 新消息接管同一会话的长任务时，插件只 abort 旧回复并不能保证 OpenClaw embedded run 已经退出。旧 run 仍可能写 session 状态，新 dispatch 随即初始化同一 session，形成残留 run、handoff barrier 与 session revision 的竞争。

### 修复方案

- 对同一账号、同一会话的独立媒体帧增加 500ms 合并窗口。窗口内到达的文字帧与媒体附件合并为一次 OpenClaw dispatch；窗口结束仍没有文字时，媒体按原顺序独立投递。待处理窗口按账号和 peer 隔离，reload/关闭时会清理定时器。
- 新消息抢占旧 Bot WS 回复时，解析 OpenClaw 6.11 的 active embedded run 并执行 abort-and-drain；排空等待有 5 秒上限，不会让新消息无限等待。连续抢占会把前一条 handoff barrier 串到下一条，避免旧清理操作与新 dispatch 交叉。
- 保留一次、仅针对 Bot WS 明确 session 初始化冲突的短重试作为最后兜底；它也覆盖插件重启后本地旧 handle 已消失、但 OpenClaw 仍残留 active run 的场景。非 Bot WS 或非精确冲突不扩大重试范围，也不重复执行已经开始的工具调用。

### 行为边界

- 单独媒体仍会在合并窗口结束后正常交付，不因等待文字而永久挂起。
- 媒体+文字只合并同一账号和 peer，跨会话消息不会串入彼此上下文。
- 新消息在媒体准备或 session prepare 阶段到达时不会伪造 drain；尚未进入 OpenClaw 的请求直接取消。
- OpenClaw 的排空接口不可用或失败时，继续使用已有的有界冲突处理和错误出口，并保留原始诊断日志。

### 验证

- 覆盖媒体后文字合并、单独媒体落单、跨会话隔离、第三条消息抢占、残留 run 排空、连续 handoff barrier、冲突恢复与 stale barrier 清理。
