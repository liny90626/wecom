# OpenClaw WeCom 插件 v2.5.110-139 变更简报

## 2026-07-17（v2.5.110-139）

### 问题与根因

本版本针对 OpenClaw 2026.7.1 生产环境的三类真实故障与一项体验需求：

1. **“思考碎片 + 投递中断”误报**（生产案例：回复只剩英文思考摘要两行 + “⚠️ 本次回复投递中断”）。OpenClaw ≥2026.6.11 在会话繁忙时会把新消息静默并入活跃 run（steer）或排入 followup 队列，回合也可能 yield 等待后台续跑；这些情形下 dispatch 都以 `{queuedFinal:false, noVisibleReplyFallbackEligible:true}` 正常返回，与“真空输出”不可区分。插件一律按失败处理，且失败提示会把最后预览（`<think>` 思考块）经 markdown 清洗（剥掉 think 标签）拼进可见正文——思考摘要因此变成用户可见的英文碎片。7.1 已逐字核实三类会话准入错误文案（`Session "X" changed/was deleted while starting work. Retry.`、`timed out draining work before reply session rollover: X`），全部在 agent run 启动前抛出，安全可重试。
2. **长任务答案静默丢失**（生产案例：长任务只出思考气泡与“任务仍在后台处理”提示，之后永远没有下文）。流窗过期（~6 分钟，errcode 846608）后 final 改走主动推送；推送若遇“结果不明”失败（ACK 超时 / 本地 8s 超时 / socket 断开——恰逢 WS 重连时高发），旧代码直接锁存 `finalDelivered` 并放弃，无重试无提示；重试链耗尽也仅留日志；新消息 activate 还会取消“一个分片都没送达”的 final 重试。多条静默丢失路径叠加。
3. **“正在思考...”久卡、新消息一发旧文字立刻蹦出**。会话被上一个活跃 run 占用时，新消息被核心排队；插件侧唯一能触发排空的时机是“下一条入站消息”（supersede-drain 中止占用 run 后核心 flush 排队答案）——因此答案总是“等下一条消息才蹦出”。
4. **需求**：“任务仍在后台处理”提示出现过早（约 5-6 分钟），希望任务处理超过 9 分钟才出现。

此外，v138 存在一处已发布的竞态：被新消息接管的 dispatch 的“冲突重试”延续体不检查 abort，可能对后继消息刚启动的 run 再次执行 drain/abort（孤儿 drain 误杀）。

### 修复方案

- **回复编排分诊**（`reply-orchestrator.ts`）：dispatch 以“无可见输出”标志正常返回时分三路——本回合有活动（reasoning/tool/进度回调）→ 静默收口（答案经后台续跑稍后送达）；无任何活动且会话仍有活跃 run → 推送“已并入当前任务”提示（附“长时间未收到请重新发送”逃生口）；真死寂 → 保留原失败兜底。被接管的 dispatch 不再产生任何合成 final。移除 core 并不识别的 `reasoningPreviewEnabled` 死选项。
- **失败提示不再泄露思考内容**（`reply.ts`）：提示只拼接可见正文（body-only 源文本），纯思考预览一律只发标准失败提示；纯思考预览不再置位 `visibleReplyStarted`（改为按 `bodySourceText` 字段存在性判定），接管时旧任务真实答案改走主动推送而非静默丢弃；被接管的纯思考 handle 收到空 final 不再向新会话推“（回复完毕）”。
- **final 投递抗丢失**（`reply.ts`）：“结果不明”推送失败改为保留投递认领并进入既有 20/40/80s 重试链，凭分片进度只补发未确认部分（防整条重复）；重试耗尽后推送一次失败提示（复用 `failNoticeSent` 一次性护栏）；新消息 activate 仅取消“已有分片送达”的 final 重试；死流上的空收口不再发必败请求。
- **接管与排空硬化**（`dispatcher.ts`）：预派发排空只做优雅 abort+settle，删除 forceClear 升级（7.1 在投递期冻结 abort 属正常状态，forceClear 会把健康 run 打成 `run_failed` 并诱发核心通用失败文案，还可能误清后继 run）；被接管 dispatch 的冲突重试路径先查 abort，杜绝孤儿 drain 误杀；排空等待统一 5 秒硬上限并补 abort 复查；冲突可重试判定扩展至 7.1 三类准入错误（对 6.11 为无害死分支）。
- **9 分钟后台提示**（`reply.ts`）：新增 `PREVIEW_EXPIRED_NOTICE_MIN_TASK_MS = 9 分钟`；流窗死亡时提示改为挂起到任务处理满 9 分钟再发，final 先到 / 被新消息接管 / 流已收口则彻底跳过；提示推送失败释放一次性锁存允许后续触发重试；定时器在收口、接管、外部活动时清理。5 分钟预览冻结与状态后缀机制保持不变（受微信 ~6 分钟流窗硬限制约束）。
- **文案**：冲突提示改为中性表述“上一轮任务还在处理中或会话状态刚发生变化，这条消息未能处理，请稍后重新发送。”，覆盖繁忙与会话变更两类场景。

### 行为边界

- 问题 3 的完整体验改善为：新消息被并入时用户会立刻收到“已并入当前任务”提示（不再是永久沉默的占位气泡）；排队答案仍需占用 run 结束或下一条消息触发排空后送达。插件不引入定时强杀活跃 run 的机制（那会误杀合法长任务）。
- OpenClaw 6.11 上真空回合（无 #100456 兜底 payload）如伴随 reasoning 活动，现以“（回复完毕）”或静默收口而非报错——生产环境为 7.1（核心自带可见失败兜底），此边界仅影响 6.11。
- “结果不明”推送重试存在有界重复风险（仅未确认分片，至多 3 次）；生产证据表明静默丢失的代价远高于偶发重复。
- Fast auto-off 后被并入/挂起的回合仍走原失败提示（罕见交叉场景，未扩大改动）。
