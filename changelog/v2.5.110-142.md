# OpenClaw WeCom 插件 v2.5.110-142 变更简报

## 2026-07-19（v2.5.110-142）

### 问题与根因

1. **长任务可能停在思考气泡或半截回复，并显示“本次回复投递中断”。** OpenClaw 2026.6.11 与 2026.7.1 都会等待 reasoning/Fast 进度回调；企微 SDK 1.0.7 又会按同一 `req_id` 串行等待 ACK。插件此前直接 `await` 企微进度网络请求，使进度侧链反向阻塞模型事件消费，ACK 变慢时可能触发 OpenClaw idle timeout。
2. **解除阻塞后，异步 reasoning、Fast 与 final 存在新的先后竞争。** 若只把回调改成 fire-and-forget，多个进度更新会并发进入同一个企微 reply stream，普通 final 也可能越过仍在等待 ACK 的进度。
3. **长任务状态可能从 `0s` 开始，且首次可见正文前的工具/推理耗时被忽略。** 旧计时从首个正文预览开始，而不是从 ReplyHandle 创建开始；长文本触发立即冻结时自然显示 `0s`。
4. **Fast 延后回复与真正失败没有完全分开。** OpenClaw 将有活动的回合转后台继续时，Fast auto-off 可能被过早当成“无正文必有中断”；被新消息接管的旧回合也可能在 abort 后继续结算失败。

### 修复方案

- **进度与模型流解耦**：reasoning/Fast 回调只登记进度并立即返回，企微 ACK 或网络延迟不再阻塞 OpenClaw 模型流。
- **单一进度通道**：reasoning、reasoningEnd 与 Fast 共用串行 Promise 尾队列；连续 reasoning 快照只保留最新待发版本，避免队列膨胀和旧思考倒序复活。
- **final 有界封口**：普通 final、合成 final 与失败路径在进入 ReplyHandle 前停止接收新进度，并最多等待 `500ms` 排空已启动进度。超过上限即取消待发旧进度；底层若仍有 ACK，则沿用既有 `5.5s` 等待和主动推送兜底，正文不会无限等待或丢失。
- **abort 后静默**：新消息接管导致 abort 后，旧 dispatch 不再发送合成 final、失败提示或迟到进度，避免污染后继会话。
- **保留 Fast 语义**：Fast auto-off 后 OpenClaw 明确转后台/延后回复时正常收口，真实无正文仍保留中断保护；Fast auto-on 可继续合法无正文结束。
- **统一任务时钟**：所有长任务状态从 ReplyHandle 创建时间计算，最低显示 `1s`；冻结预览在健康流内每 15 秒推进，流窗口失效后继续沿用 v141 的 9 分钟起每分钟主动状态。
- **区分错误来源**：OpenClaw 模型 idle/turn timeout 使用“模型响应超时”提示，不再伪装成企微投递中断。该提示仅用于真实错误分诊，根因修复仍是解除进度回调背压。

### 源码对照

- OpenClaw 6.11/7.1 的 `wrapProgressCallback` 都会等待渠道回调，普通 block/final 通过 dispatcher `sendChain` 串行并在结束前等待 idle。
- 官方 Discord 在 final 开始后停止新进度并先 flush draft；飞书用 `partialUpdateQueue` 串行更新并在 close/discard 前排空；WhatsApp 将最终正文交给 durable delivery pipeline。
- 本版采用相同生命周期原则：进度单一所有者、final 明确封口、关闭前有界排空、abort 后禁止旧更新复活，同时保留企微流窗口和 ACK 约束下的主动推送兜底。

### 验证

- 先稳定复现“reasoning ACK 阻塞触发 idle watchdog”“普通 final 越过在途 reasoning”“长任务显示 `0s` 且计时重置”，再验证修复后不复现。
- 真实 `createBotWsReplyHandle` 跨层模拟覆盖：ACK 在 20ms 后恢复走原流 final；ACK 持续阻塞则 500ms 封口后经既有等待转主动推送完整正文。
- 全量 Vitest：42 个测试文件、405 个测试全部通过；核心回复套件无缓存复跑 170 个测试通过。
- `npm run build`、`npx tsc --noEmit`、`npm run verify-dist`、B1/B2/B3 检查与 `git diff --check` 全部通过。
