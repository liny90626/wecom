# OpenClaw WeCom 插件 v2.5.110-143 变更简报

## 2026-07-22（v2.5.110-143）

### 问题与根因

1. **正常暖会话可能在模型派发前卡住，最终出现会话准备超时。** v142 会等待 OpenClaw `recordInboundSession` 跟踪的全部 metadata task；这些 best-effort 写入共用 agent 级串行队列，任一历史写入卡住都可能阻塞后续正常消息。v118 不等待 metadata task，因此没有这一退化。
2. **长任务已有进度后，真实 OpenClaw 错误 final 会被误装成成功回复。** 当原流窗口失效时，`LLM request failed.`、`LLM request timed out.` 或通用 run failure 仍进入成功 fallback，可能显示“继续输出”并追加“（回复完毕）”。这不会制造模型错误，但会掩盖任务实际未完成。
3. **prepare timeout 会泄露内部错误。** 用户可能直接看到 `WeCom WS reply failed: WeCom inbound session prepare timed out...`，无法判断该消息尚未开始模型处理。
4. **发布前生产依赖审计发现两个高危项。** `fast-xml-parser 5.10.0` 与企微 SDK 间接依赖的 `axios 1.16.0` 均有已发布安全公告。

### 修复方案

- **暖会话立即派发**：继续调用 OpenClaw 正常登记入站会话，但不等待其 best-effort metadata task；后台 task 立即挂接 `Promise.allSettled`，失败不会形成未处理拒绝。
- **冷会话有界等待**：首次会话保留初始化顺序保护，但最多等待 1 秒；超过窗口记录 `inbound-session-metadata-deferred` 后继续派发，不再让 metadata 队列占满 60 秒 prepare 上限。
- **失败 final 保持失败语义**：`isError` final 的流失效接续改为“任务未完成”，不再使用“继续输出”，不再追加“（回复完毕）”；分片进度和既有有界补发身份保持不变。
- **友好 prepare timeout**：明确提示“会话准备超时，本条消息尚未开始处理，请稍后重新发送”，不泄露 agent/session 或内部英文错误。
- **安全补丁**：`fast-xml-parser` 升至 `5.10.1`，SDK 保持 `1.0.7`，其允许范围内的 `axios` 升至 `1.18.1`。没有增加直接 axios 依赖。
- **测试稳定性**：watchdog 用例改为跳转 fake system time 后触发一次 interval，不再执行 240 次无意义刷新；生产逻辑未改变。
