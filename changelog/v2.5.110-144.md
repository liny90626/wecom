# OpenClaw WeCom 插件 v2.5.110-144 变更简报

## 2026-07-23（v2.5.110-144）

### 问题与根因

1. **OpenClaw 已完成回答并确认外部投递，企微仍可能只显示“本次回复投递中断”。** OpenClaw 6.11/7.1 在 message-tool source reply 有真实可见投递证据后设置 `observedReplyDelivery`；企微 SDK 1.0.7 的主动推送 Promise 也只在服务端 `errcode=0` ACK 后 resolve。插件此前只停止旧 ReplyHandle 的预览定时器，没有按“外部 final 已交付”结算原流。
2. **旧流会错误启动多段补发。** orchestrator 最后仍发送普通空 final；它会合并旧进度，原流过期返回 `846608` 后转入 `20/40/80s` final retry。补发耗尽后再主动推送“本次回复投递中断”，形成答案成功但旧投递链反向报错的矛盾状态。
3. **旧流错误可能覆盖成功结果。** 即使 external delivery 已观察到，旧 block/tool/final 的 callback error 或 failed count 仍可能先进入失败路径，产生相同误报。

### 修复方案

- **外部 final 统一结算**：正常 dispatch 返回和“外部投递后核心再抛错”两个出口，都复用既有 `wecomExternalFinalDelivered` 语义；ReplyHandle 只在本地结算并静默关闭可用原流，绝不重新主动推送旧进度。
- **成功优先于旧流错误**：`observedReplyDelivery` 在 block/tool/final 错误分诊前统一早返回，已提交答案不会被旧流遗留状态覆盖。
- **删除冗余条件**：外部成功早返回后，删除 Fast/no-visible 分支中两个多余的 `!observedDelivery` 判断，末尾恢复普通 close。没有为修复增加新的状态字段、timer、轮询、重试或网络等待。
- **保留真实失败保护**：只有 OpenClaw 明确证明 source reply 已外部投递时才静默结算；未观察到投递的 final 继续使用既有流式发送、主动推送和有界失败提示。

### 复现与验证

- 修复前使用真实 ReplyHandle 时序稳定复现：连续两轮均先显示进度，随后原流返回 `846608`，各自执行 `20/40/80s` retry 并各产生一次“本次回复投递中断”；第三轮恢复正常 final。
- 修复后同一模拟中前两轮不再启动 stream fallback、final retry 或 fail-notice，第三轮普通 final 仍正常交付。
- 补充 callback/result 两种来源下 block/tool/final 六类旧流失败矩阵，以及 Fast auto-off 前后观察到投递、投递后核心异常等回归。
