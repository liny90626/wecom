# OpenClaw WeCom 插件 v2.5.110-145 变更简报

## 2026-07-24（v2.5.110-145）

### 问题与根因

1. **外部 final 已成功，旧 final retry 仍可能继续运行。** v144 已让 `observedReplyDelivery` 优先静默结算，但 ReplyHandle 中此前排队的 retry timer 不会自动取消；若 retry 已在执行，它失败后还可能重新安排下一轮，造成成功答案后迟到补发或误报。
2. **placeholder keepalive 会跨账号或跨会话类型串扰。** 全局 keepalive 注册表只按裸 `peerId` 分组；同 userid 使用两个企微账号，或 direct userid 与 group chatid 相同时，一边开始回复会错误停止另一边的保活。
3. **热重载旧实例可以删除新实例。** Bot WS push handle 与 account runtime 的注销只携带账号，不校验调用者是否仍是当前注册对象；旧 adapter/monitor 较晚执行 `stop/finally` 时会清掉已经接管的新句柄。
4. **来源查询可能跨账号，活跃来源快照也可能被错误淘汰。** 已知 `accountId` 的查询在本账号 session 未命中后仍先读取 loose session，可能拿到另一账号的 Bot WS/Agent 来源；有界注册表又把 `Map` 插入顺序当作刷新顺序，但 `Map.set(existingKey)` 不会移动键，持续活跃的老快照反而最先被删除。

### 修复方案

- **终态同步取消旧 retry**：external-final 分支复用既有 `obsoleteFinalRetry` 与 `finishPendingFinalRetry`，同时取消等待 timer，并阻止在途 retry 失败后重新挂起。没有新增状态字段或重试机制。
- **keepalive 完整作用域**：注册、通知和清理统一使用账号、`direct/group` 与规范化 peer 组成的既有 `replyPeerKey`，不同账号和会话类型互不干扰。
- **注销校验注册属主**：push handle 与 account runtime 的 unregister 必须传入原注册对象，只有当前值仍与它相同才允许删除；SDK adapter 保存自身 handle 身份用于正常停止和热重载交接。
- **source registry 严格账号边界**：调用方提供账号时，只按该账号的 session/sessionId/conversation 查询，不再回退到其他账号的 loose 记录。
- **正确维护最近顺序**：快照更新统一先 `delete` 再 `set`，让容量淘汰真正保留最近刷新项。
